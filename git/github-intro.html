Intro to Git, GitHub, and CLI

In this lesson you will:
  1. Create a GitHub account
  2. Create a GitHub repository
  3. Install Git onto your computer
  4. Learn the basics of navigating a CLI
  5. Push a project up onto your repository

GitHub Account

What is GitHub?

GitHub is a very widely used platform for sharing projects and collaborating with other developers. Head over to GitHub!

Click "sign up" if you don't already have an account. Sign in if you've already made one.

Fill in the fields (be sure to verify your email address if you're making a new account!) and submit
Creating a Repository

After you've signed into your account, you'll be on your dashboard. Think of this as your account's homepage

On the left side of your dashboard, you'll see a list of your repositories. Repositories are containers that store your project files and their edit histories.

Next to the word 'Repositories', look for a green button labeled "New" and click on it.
You'll see fields for your repository's name, a description of it, its privacy, and a choice of adding a readme file.

Readme files usually contain information that the developer wants readers to know about their project. They tend to be written using Markdown, a language that's very similar to plain text and HTML. Learn more about Markdown here

We'll leave the privacy as public, and we'll want to check the box labeled "Initialize this project with a README".

We can use this option to allow others to clone our repository, so that they can collaborate with us.

Once everything is filled out, click the "create repository" button. Congratulations, we've made our first repository!
Git installation/Getting started with our CLI

CLI stands for "Command Line Interface". It's a more direct way to interact with your machine than a GUI(Graphical User Interface), which many users are more accustomed to.

On Windows Machines, we'll be using Powershell as our CLI. For OSX and Linux, we'll use Terminal.

Git itself is a version control system, and it makes up the backbone of GitHub.
Version control is a system that tracks changes in files over time, so that a user can change projects over to different states without having to manually edit their files each time.

If Git isn't installed, we'll need to install it. Head to the Git downloads page and download the version that matches the machine you're using.

The site will usually be able to auto-detect your operating system(OS), the software that controls your computer's basic funtions.

Open up the installer and follow the steps, the default settings are fine. When asked if you'd like to install Git Bash, select "no". When asked to use Vim as the default editor, choose Atom or VS Code, whichever you prefer.

Open up your CLI.

If you aren't sure where to find your CLI, try typing its name into the search bar on your machine's desktop.

On Windows, we'll need to edit our PATH variable to use git commands with Powershell.

Use Windows Explorer to navigate to your Git installation (it's probably in C:/program files).

Inside of the Git folder, find the mingw32 (or mingw64) folder

Inside of the mingw folder, find the bin folder.

Copy the full path of the bin folder.

Right click the folder -> properties -> location

Open up control panel

Navigate to System And Security -> System -> Advanced System Settings

Select the "Advanced" tab

Click on environment variables

Select the path variable by double-clicking on it.

We're going to be editing the "user" variables version of PATH, so make sure you've selected that one, NOT system variables.

Click on the "new" button, and paste your bin directory into the text box.

Once you've done that, go ahead and confirm your changes.

Open up PowerShell and type 'git' into the console. Hit enter.

If a list of options for Git appears, you're all set!


Feel free to read through the options list! It may come in handy
CLI basics

We use commands to tell our CLI what we want it to do.

To move around our machine's filesystem, we'll use the "cd" (change directory) command
To use the cd command, we'll type "cd" into our CLI, followed by the path of the place we want to go (ex. If I want to go to my default documents folder on windows, I'll type "cd documents") and hit enter.

If the folder you're trying to access has a space in its name, you'll need to put quotation marks around it (ex. cd "my folder"). If you need to go back up a level in the diretory, you can run "cd .." to do so.

Other useful commands are "ls", which will show you all the files in your current directory, and "pwd", which will tell you what directory you're currently in
Initializing Git

Now that we've made our repository, we'll need to create our files and initialize Git inside of our project.

In your CLI, navigate to your project's root folder.

Your root is the bottom level of your project, that all other files and folders in the project are contained inside of.

In your root, run "git init". This will initialize Git within your project.

Next, we'll need to point our local repo at its remote counterpart.

On your GitHub repository page, click the green button labeled "clone or download".

Make sure your prompt says "clone with https". Copy the link it gives you in the box.

In your CLI, we'll type "git remote add origin" followed by the link we just copied.

Once the confirmation pops up, we're all set to start pushing!

Commiting and pushing

When it comes to pushing code to GitHub, we can usually follow an outline that goes as follows:
  • git add .
  • git commit -m "commit message"
  • git push

Let's break down what's happening here. This is the equivalent of saving a word document, but with code. Unlike saving a word document where we just click a "save" button, using git is distinct because it allows us to use version control. We can specify what files we want to update, what version of the project we want to update them on, and more!

The first step to saving our code is to stage it on our local machine so that we can push the changes us to the remote (on GitHub) version of the project.

The command that does this is "git add .". The period at the end of the command tells git to look through all of our files for changes to stage for saving. We can even specify as far as individual portions of files, but we'll cover that a bit later.

Side Note: Git is actually its own language. You're learning another language! Yay!

After we've staged our changes (using "git add .") we need to "commit" them
The concept of the "commit" can be a little hard to wrap your head around at first. It isn't actually saving the project to our remote version in github yet.

What it IS doing is bundling up our staged changes in a nice way that allows us to better track and manipulate changes to our codebase. Think of this as more of a labeling process. First you staged your code by grabbing all of the changes that were made when you ran the command "git add .". Now you'll label those changes so they're easy to find in all of the changes that have happened and will happen to the project's code.

As mentioned, part of this commit process is labeling your code. In other words, adding a message.

Our Git command reflects that. To commit, enter `git commit -m"a message describing what changes were made to your code"` into your CLI. The "git commit" part of this is fairly straightforward. The "-m" part stands for message, and it will stamp the message we put inside of the quotation marks on all of the files we staged in our project.

Now that we've staged our changes and packaged them up nicely with a commit message, we actually have to push them up to the remote (on GitHub). We do this by running the command "git push". Now our changes should be on GitHub!
Bonus: Comparing local and remote
What if we're not sure where our local branch stands in relation to remote? We can run "git branch -av" To see a list of recent commits!
