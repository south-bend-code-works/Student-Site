Objectively Speaking

We need to take a moment to recognize the JavaScript elephant in the room: JavaScript is weird. It was designed to be a scripting language for the web with a Java-esque syntax, which itself was inspired by C. However, it is very unlike most if not all C-based or C-inspired languages. And it’s also totally different from Java.
The name JavaScript was a blantant attempt at gaining popularity by riding on the coattails of the up-and-coming Java that was all the rage at the time. It worked.
All of that is to say JavaScript does not behave like most other languages, objectively speaking, though it can seem like it does, especially given the work that has been done on the language to make things easier for anyone who use a classic class-based object-oriented languages. This work comes in the form of syntactic sugar.
Syntactic sugar is any kind of trick that a programming language has added to make it easier to deal with, just like a spoonful of sugar makes the medicine go down. It usually hides needless complexity making it better for most tasks. Think of having a*b instead of multiply(a,b).
In last lesson, we introduced the native JS Object using object notation. We can use functions to build objects like that, a process called prototyping — JavaScript is a prototype-based language. Or we can take advantage of that syntactic sugar and build them using classes like one might in Java or C++ or C# or Objective-C or Kotlin or Swift…virtually any other object-oriented language. So that’s what we’ll do.
Object-oriented programming (OOP) is another programming style like functional programming or procedural programming. The main difference is that instead of having only global and local variables, we can have variables (the properties) and functions (the methods) attached to objects. As mentioned before, each style has its benefits and flaws — most projects end up mixing styles together.

Blueprints

So what even is a class? It often gets described as a type, but it has notable differences. The word class actually comes from type theory to mean structured data with methods for manipulating it.
A good way to think about a class is as a blueprint to build an object and define its associated characteristics and actions, variables and functions, respectively. We can think of these as the nouns and verbs that tell us about the object and what it does.

One of our smiley and awe faces nouns is their position. What do you think are other nouns of our face dudes?

In addition to position, their nouns are size and speed.

What about the verbs?
Hover for a while to reveal a hint
The verbs are drawing, moving, animating, bouncing off walls, and keeping in bounds.
Now just like a blueprint for a house doesn’t usually specify color or location or other non-structural details, the class doesn’t usually tell us the values of variables either. That gives us the flexibility to define them when they are made.
Making objects is known as instantiation. The made object itself is known as an instance of the object/class. The difference between an object and an instance is like the difference between computers and this computer. Informally though, we’ll often say object when we mean instance and clarify when context doesn’t make it clear.

Let’s make the blueprint for our faces. We are going to put it at the very bottom of sketch.js. When we are finished, everything but the main setup() and draw() function will be included in a class. Give yourself a bunch of lines of space at the bottom.

Next, we just declare a class almost exactly like we declare a function. Put class Ball {} on a new line. We are not going to define the faces just yet, only the way the shape is going to act — a name like Ball seems good because it bounces.
Notice that the B in Ball is capitalized. We are going to PascalCase our class names to distinguish them from the instances of the class. Also, the class name itself is a single thing because we make one ball at a time.

Inside the block (between the curlies) we are going to put all of the associated variables and functions, so give yourself some new lines between the brackets.

The first function we are going to define is the constructor(). For a ball, that is the properties that both of our faces share: two for position, one for size, two for speed. Like last lesson where we consolidated our parameters into one object that contained all of our values, we can do that for both position and speed. Put constructor(pos,size,speed) on the first line in the class.
Since we have circles and their width is their height, we only need one value for size. However, if we had ellipses or rectangles, we could still put one size variable like we will do with pos and speed and give it the same kind of treatment using a pair of values, whether the built-in vector or a small class we make ourselves.

It will also be useful to define a property for it’s appearance, whether a color or a face. Let’s call it look and add it as the first parameter: constructor(look,pos,size,speed).

Now make a block for implementation just like any other function. Inside these curlies, we are going to set our parameters to our properties.
In JS, the way that we tell what is a property is by attaching it to an object like we did with awe.x. But what object exactly? The blueprint is not an instance of an object. The object does not yet exist. But we are talking about this object when it does exist at some point in the future. So in JavaScript, we use the keyword this, like this.x.

Put this.look = look; on the first line. This assigns look to the property this.look. Set the other three properties to their corresponding parameters.
Another oddity of JavaScript — we don’t ever declare the properties like we have done variables in the past. As soon as we put this.whatever — it exists for this object. We can even add such a property to an already instantiated object — a feature of the prototyping paradigm. The drawback of this is that we always have to put this dot in front of our properties. This is not the case in most other languages.

Vectors and Values

We must take another little detour to understand what our new pos and speed properties are. These are vectors, which is just a fancy term for a set of coordinates that have special methods defined for arithmetic operations like add and multiply.
For position, it is indeed just (x,y), but for speed, it has this extra vector-y meaning: It’s not just (x,y) but an arrow from an origin like (0,0) to (x,y). That is, it also tells us a direction, a trajectory — perfect for the likes of a velocity.
What we need to assume is that pos and speed are vectors and setting our properties to them makes them vectors as well. But remember all that pass-by-value and pass-by-reference talk? p5.Vector is a class and an instance of it is an object — so which kind of passing will it be?
Hover for a while to reveal a hint
Objects are pass-by-reference. That means that the property will be the same thing as the object. Both will point to the same spot in memory — that is, the same “burrito”.

Let’s make sure that we have new copies of the vectors. One way is to explicitly create a new vector. Put this.pos = createVector(pos.x,pos.y); in place of what you did before.
You can think of createVector() as a shortcut for the p5.Vector constructor. Notice that it takes two values, x and y. These are the properties of the vector in two dimensions. (In 3D, we also have a z for the forward–backward (zoom) dimension, but we leave it undefined in 2D.)

Seeing how we addressed the x and y of the parameter, how do you think we will address our property in the class?
Hover for a while to reveal a hint
Since the x of the parameter is pos.x, in the class we will put this.pos.x. It’s the same for y: this.pos.y.

What about outside of the class? What if our instance of Ball was just called ball?
Hover for a while to reveal a hint
Instead of this, we put ball: ball.pos.x and ball.pos.y.

The second way to get a copy of a vector is to use the copy() method: speed.copy(). Recall that methods are just functions attached to objects. Set this.speed to it.
There is an added bonus to creating a class versus just declaring a JS object like we did last lesson: Within the constructor we can use any of the functions and environment variables that we can use inside setup(). That means width and height, declaring new objects, and using p5’s random() function that we will use quite a bit.

Now we need to build our objects. The code is pretty simple: 
ball = new Ball(
          stringOrColor,
          positionVector,
          sizeValue,
          speedVector
        );
Notice that when we declare this ball, we put the name of the class with parentheses after it as if it were a function. That’s because it is a function. It’s the constructor that we defined. Instead of putting something like Thing.constructor(stuff), we save some typing: Thing(stuff).
Because each parameter we give to the constructor is going to be pretty long, each has been put on its own line with the parentheses resembling the curly braces for a function body. This makes it much easier to tell which parameter is which. We don’t care about using extra space, so don’t be stingy.
For the look, we will be putting the kind of face like "smiley", the kind of color in quotes like "red", the hexcode in quotes like "#0000FF" (blue), a color variable we defined like skyblue, or anything else that fill() will take. This gives us a lot of flexibility. We are going to choose the first for this demonstration.
For both position and speed, we need to create a couple vectors. It’s the same createVector() function that we used above for both. But instead of choosing the values ourselves, we are going to throw some randomness in for good measure.
We’ll have something that looks like this: 
smiley = new Ball(
          "smiley",
          createVector(
            random(100, width - 100),
            random(100, height - 100)
          ),
          random(50, 100),
          createVector(
            random(2,8),
            random(2,8)
          )
        );
See how we have expanded the parentheses for both Ball() and createVector(). You don’t need to do that, but once you get used to it, it’s a bit easier to tell what is what.

An About-face

Instead of fixing that code in Ball(), we can take advantage of something known as inheritance. Think of it like a child inherits features from a parent. It’s a way that we can extend this less defined parent class — a Ball that has no use for sunglasses — into something more concrete — a Face that might use them. We extend Ball using the keyword extend.

Below the Ball class, we will create a new Face class like so: class Face extends Ball {}.
Just like in Ball, we need a constructor. We can just copy and paste the one from Ball and add the offset property, but that defeats the purpose of inheritance.

We do, however, want the parameters to be the same. So in the body of Face, put constructor(look,pos,size,speed) {}.

Now instead of redeclaring look and friends, we can just call the parent constructor with the universal name for parents like Robert “Totally Average” Parr (a.k.a. Mr. Incredible): super(look,pos,size,speed);

Then we can create our offset property and set it to height. What do we need to do to declare a property?
Hover for a while to reveal a hint
A property must be attached to an object which inside the class implementation is the keyword this. this.offset = height;
