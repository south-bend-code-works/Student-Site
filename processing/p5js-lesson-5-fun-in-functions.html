Find the conditional statements that keep smiley in bounds.
Here we can streamline our code by using a built-in function called constrain(). What it does is take a value in a variable and check to see if it is within the bounds we have given it and then returns a value within that range.
The signature of constrain() looks like this: 

function constrain(value, lowerBound, upperBound) {
  // perform some magic
  return constrainedValue
}
It takes three parameters and returns a value based on them after they have been put through some mathematical magic.
A function signature is the name of the function, the types of parameters it takes, and what kind of value it returns when it finishes, including not returning anything. Most C-based languages have very informative signatures, but Javascript, like most popular scripting languages, doesn’t tell you what kind of values it takes or returns in its signature.

What do we mean by return?
Generally speaking, there are two kinds of functions: those that just do something, also called procedures or subroutines, and those that take some value and transform it, giving a value in turn — the classic mathematical notion of function which take inputs and give outputs.
This second one is what constrain() is —Â we give the function something and it gives us a value in return. We give it values as parameters and it gives us a return value.
Since it gives us a value back, we need a variable to catch it. That just means setting a variable to its result. That variable can be a new one, preserving the value of the one used as a parameter, or as it will be in this case, the same variable, clobbering the previous value.
Believe or not, clobbering is the technical term for overwriting contents of memory, often unintentionally. It is usually followed by some form of facepalming and headdesking.

Congratulations — you just refactored code. You have changed it and rearranged it to be easier to understand. But you’re not done yet.
Refactoring code is an often maligned but necessary part of the programming process. Writing code is often an adventure into the unknown. You may not know a good way to solve whatever new problems you encounter, let alone the best way. But the only way to start such an adventure is to start, try out ideas, solve problems, and then spend the time to clean up your code and write comments about what it does.
You may find better solutions through this process, or you may not, but one thing is certain: Your future self and anyone else who has to deal with your code will thank you.

Form Follows Procedure
The functions we have demonstrated the procedural programming style. Even though most languages today don’t distinguish between pure functions and procedures — they are all lumped under function — we can and should note the difference.
Not only do none of the above return a value, they don’t even have parameters that could tell us what they might be changing. And crucially, all of them manipulate the global variables defined at the beginning. Look up procedure in the dictionary, and you’ll find the spitting image of these guys.

Why does this matter?
Notice that separate blocks of code use rad, but you will only need to define it in the first function that uses it. This should make you a little uneasy.

Functionally Functional Functions
This kind of code where one function changes the way another function behaves is called side effects by proponents of functional programming. Side effects usually aren’t good, and this case is no different. A function that changes how another function is going to act can lead to many hours rooting out errors.
For a small program like this, side effects aren’t much of an issue. Changing a small set of global variables with functions that don’t tell you they are changing them isn’t a big deal.
In fact, because it is easy to keep most of this stuff in mind, procedural programming is kind of great. How simple was it to just cut and paste into the function body and then just call it?
But as programs get more complex and debugging consumes more and more time, it’s a really good idea to have functions that you don’t have to look inside of to understand what they might affect the state of your variables.

function drawAwesome() {
  push();

  scale(w / 300);

  strokeWeight(4);

  fill(yellow);
  circle(0, 0, 300);

  fill(white);
  arc(-80, -25, 88, 100, 157.5, 22.5, CHORD);
  arc(40, -28, 100, 100, 157.5, 22.5, CHORD);

  fill(black);
  circle(-62, -52, 30);
  circle(64, -52, 30);

  noStroke();
  fill(white);
  circle(-70, -60, 10);
  circle(56, -60, 10);

  fill(plum);
  arc(-6, 24, 200, 200, 0, 180, CHORD);

  fill(pink);
  beginShape();
  vertex(62, 96);
  bezierVertex(42, 68, -16, 80, -5, 123);
  bezierVertex(17, 123, 42, 114, 62, 96);
  endShape();

  stroke(black);
  noFill();
  arc(-6, 24, 200, 200, 0, 180, CHORD);
  pop();
}
