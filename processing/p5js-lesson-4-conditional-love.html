The condition is the smiley is touching the wall
A condition is a statement that has a truth value — it’s true or not true, also known as false. These are known as Boolean expressions.
George Boole, one of the grandfathers of computing, was a logician and philosopher in 1800s who came up with a way to manipulate things which are true or false as if they were numbers, giving us conjunction — AND — which is kind of like multiplication, disjunction — OR — which is kind of like addition, and negation — NOT — which is kind of like, well, negation. This logic, now known as Boolean Algebra, is one of the foundational elements of computers that use true or false, yes or no, on or off, 0 or 1.
Note that binary is not the same as Boolean, it’s a way of representing numbers as a series of yeses and noes.
Consider these two sentences:
  • This is the same as that.
  • This is different from that.
One of them is guaranteed to be true and the other is guaranteed to be false — as long as we are judging both by the same criteria.
In Javascript, we can express those sentences like so:
  • This == that where == means is equal to
  • This != that where != means is not equal to

Now if I told you red is the same as green, would you agree? What if I said, if red is the same as green, I’ll eat my hat. Am I eating my hat right now?
(You would be incredulous if I told you it was tasty.)
That sentence is a conditional: a set of actions that depend on a condition being true.
In Javascript, we write a conditional like this: 

  if ( /* condition */ ) {
      /* a set of actions */
  }



  if (red == green) {
      eat(hat);
  }

For this last bit, we need to check to see if a few conditions are true at the same time. We can do this a couple different ways: the first by nesting if statements, the second by chaining conditions with Boolean operations like AND && , OR || , and NOT ! .
Both are demonstrated in this code: 

  if (mouseIsPressed) {
    if (mouseButton == LEFT) {
      if (offset > 0) {
        offset -= 2;
      }
    }
  }
  if (keyIsPressed && key == ' ' && offset < height) {
    offset += 2;
  }

In this case, both will work just fine.
In fact, the machine code they get turned into is likely identical. The first one has a bunch of unnecessary curlies that may look foreboding; the second one has a line that runs a little long for my taste; there is a third way that combines the two, and a fourth, fifth, and sixth way that eliminates the curly brackets from the first three.
But as soon as we add other conditions, like another key or button to be pressed with other statements to be executed — the nested way with the curlies wins out. Nesting gives you a huge amount of flexibility and the curly brackets make sure that new lines you add won’t mess up your code.

So let’s keep the nested style code and comment out the chained one.

To get this guy to bounce, we need to change the speed that each direction is moving. So, for this to work, we need our speed to be a variable.
After we declared offset, add xSpeed and ySpeed, capitalizing the S.
This is called camel case, so named because it has humps, like a camel. If the first letter is capitalized, it’s known as UpperCamelCase or PascalCase. If the first letter is lowercase, lowerCamelCase or dromedaryCase. (A dromedary is a camel with one hump.)
Camel case is the style that p5 uses, and to write clean and readable code, it’s best to adopt the style of the language or framework we are using.
Be sure to capitalize the S in speed everywhere, because xSpeed is not the same as xspeed.

Now in setup(), set the values of these variables to one.

Last, we need to change those constant values added to our position variables to xSpeed and ySpeed.

Run your code to make sure our guy still runs to the lower right of the canvas.

Because the detection of the walls are mutually exclusive and we want the same thing to happen, we’ll chain these conditions together.
Below the code that keeps our guy in bounds, paste this: 

  if (x + xSpeed - rad < 0 ||
      x + xSpeed + rad > width) {
    xSpeed *= -1;
  }
  if (y + ySpeed - rad < 0 ||
      y + ySpeed + rad > height) {
    ySpeed *= -1;
  }

Notice how it’s similar to the code to keep our guy from leaving the canvas? What is different about the conditions?
Why do you think that is?

Since a bounce is a reflection, we multiply the speeds by the inverse, -1, so that instead of adding one, we subtract from our position, making our guy go left and up, respectively, then flip the other way when it hits another wall.
That’s pretty cool.
