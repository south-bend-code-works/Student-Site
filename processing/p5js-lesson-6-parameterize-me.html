Pairing Procedures with Parameters

Remember how we created variables so that we didn’t have to change every single instance of a value any time we wanted to update it? We can do something similar regarding changing the input of a function.

Variable Parameters
Let’s take circle() as an example. It has three parameters: x position, y position, diameter/width. Our characters are circles. We can add the same parameters to our functions.
We have used three global variables, x,y,w to define each face’s position and size, but no circle() inside either face takes those parameters. What gives?
Through the magic of using built-in functions, we don’t need to make every detail in the drawing depend on those variables. Instead, we have put our x,y and w in translate() and scale(), turning our absolute coordinates and sizes of every aspect of each face into relative ones.

Take a look at this function implemetation with the original values:
We have moved the center of the canvas with translate(). That moves the origin (0,0) to the coordinates (x,y). We have also changed the zoom of the canvas using scale(). Since the original face was drawn 300 pixels wide, the scale factor w divided by that original size 300 keeps everything in proportion.
The push()-pop() pair keeps traslate() and scale() from affecting any of the other drawings we may have.

We have successfully disentangled smiley and awesome, but smiley doesn’t move anymore. We are not changing its coordinates every frame like we are with the awesome face.
To do that, the quick and dirty way is just to add more variables like x2Speed and y2Speed and then add them to our animate() function as parameters with our position and then change the position. It seems like that would work, but there is something funny about the parameters that are plain number values — known as primitives.
A primitive is simple data type like Booleans (true/false), integers (whole numbers), and floating point numbers (decimal fractions). They don’t have any extra data attached to them, so they’re very small in size and easier for the computer to copy into faster memory rather than use it in place.

For Lack of a Reference Parameter

Sometimes when programming, things are easy — they just kind of work — and we don’t really ask questions. And then there are times like this. In C-based languages, parameters come in two varieties: pass-by-value and pass-by-reference.
Rather than explain the difference, how about story time?

A Tale of Two Burritos

It was the best of times. I sat down at the Flying Burrito Monster, ready to dig into an artisanally crafted feast for the senses, when a shadow darkened my meal. “What kind of burrito is that?” It was my older brother. “Beans and rice,” I said. “Looks good. I’ll go order one,” he said.

It was the worst of times. I sat down at the Flying Burrito Monster, ready to dig into an artisanally crafted feast for the senses, when a shadow darkened my meal. “What kind of burrito is that?” It was my older brother. Before I could stop him, he snatched the burrito off my plate and took a big bite. “Beans and rice,” he mumbled. “Tastes good. You should order one.”

In the best of times, dear brother got the answer to his question without having to consume the burrito. That’s pass-by-value — a copy of the variable is used. He went and got his own version of the burrito and didn’t touch mine.
In the worst of times, dear brother got the answer by consuming the burrito. When he finished, the burrito was irreparably changed. That’s pass-by-reference — the variable itself is changed.

Do you see where this is going? We want brotherChecks(myBurrito) in the worst of times. But in JavaScript, we have brotherChecks(myBurrito) in the best of times whenever the variable is one of these primitives. We need brotherChecks() to change myburrito, but all brothers are really nice when given simple value variables in JavaScript.
We have two options: turn myBurrito into a JavaScript object or use the functional programming style to explicitly reassign the values. With one exception, we will do the first.
Objectively Simple
In most programming languages, simple variables will be pass-by-value by default while larger data structures like objects and arrays which can hold millions of properties and elements are pass-by-reference. The reason for this is pretty straight-forward: Copying a single value costs nothing but a tiny bit of space; copying millions of values costs millions of times more.
Since JavaScript has no way to make primitives pass-by-reference, we can use the native JavaScript object which is pass-by-reference.
It looks something like this: 
objName = {
  key1: value1,
  key2: value2,
  key3: value3
}
The keys are the names of the properties aka attributes aka fields aka member variables. (We’ll stick with properties for JavaScript.) They are just variables with fancy names that cannot be used on their own. Simply putting key1 will not give us the property up there. To address it, we have to call it by its full name: objName.key1.
So with smiley and awe it can look like this: 
// smiley/sunglasses attributes
  let smiley = {
    x: 100,
    y: 100,
    w: 100,
    xSpeed: 3,
    ySpeed: 4,
    offset: 400
  }
  // awesome face attributes
  let awe = {
    x: 300,
    y: 300,
    w: 100,
    xSpeed: -5,
    ySpeed: 2
  }
Instead of having to put the arbitrary x1 and x2 or the more-descriptive-but-still-arbitrary smileyX and aweX or whatever naming convention we choose, we have these variables literally attached to the objects that use them. Pretty convenient. What is awe’s position? (awe.x,awe.y). What’s smiley’s width? smiley.w Makes sense, right?

One Parameter to Rule Them All

We have taken a bit of a journey, but it’s time to get back to the task at hand. We can keep having to call every function with individual variables — our properties attached to their objects — ending up with a rather crazy amount of parameters, but since our variables are attached to our new fangled objects, why don’t we just call the functions with the objects themselves? 
drawsmiley(x,y,w);
drawSmiley(object);

And then instead of just x inside the function, we can add object. in front of it: object.x. Because it can get tedious to write out object all the time, we’ll just call it ob.

function drawSmiley(ob) {
  push();
  translate(ob.x, ob.y);
  scale(ob.w / 300);

  strokeWeight(4);
  stroke(black);
  fill(yellow);
  circle(0, 0, 300);

  fill(black);
  ellipse(-50, -15, 30, 40);
  ellipse(50, -15, 30, 40);

  arc(0, 20, 200, 150, 20, 160);
  fill(yellow);
  arc(0, 15, 220, 130, 19, 161);
  pop();
} // end of drawSmiley()
That’s pretty cool. However, this is going to get tedious to replace everything in some of the other functions — bounceOffWalls(ob) — I am looking at you.

The great thing about variables is that you can assign them to other variables, like so: 
function bounceOffWalls(ob) {
  // set the properties to local variables
  let x = ob.x;
  let y = ob.y;
  let xSpeed = ob.xSpeed;
  let ySpeed = ob.ySpeed;
  let rad = ob.w/2;

  if (x + xSpeed - rad < 0 ||
    x + xSpeed + rad > width) {
    xSpeed *= -1;
  }
  if (y + ySpeed - rad < 0 ||
    y + ySpeed + rad > height) {
    ySpeed *= -1;
  }

  // change the speed properties
  ob.xSpeed = xSpeed;
  ob.ySpeed = ySpeed;
} // end of boundOffWalls()
As long as the values we want to change are reassigned like we’ve done with the speeds, it will work as intended.
It may occur to you that this bit of laziness might be wasteful and slow things down. Eventually, that may be true — but now is not the time for optimizing our code. First we get it working. We may be accruing some technical debt, but we also have to get the job done.
info: Technical debt is used to describe the refactoring and rewriting of code that may need to happen in the future. But at this point, we still don’t which functions will be necessary for our final program. As the legendary Donald Knuth popularized: Premature optimization is the root of all evil. Though they often don’t add the part where that’s only true about 97% of the time. We won’t worry about the other 3%.

In animateShades(), we are doing something a little different. We’ll give the function one parameter and instead of changing the parameter, we return a value can “catch” it with the variable we want to change, which in this case is the same as the parameter.

Give animateShades() a single parameter called offset.

Tossing a value back is simple — we just return the offset after the if-statments as the last line in the body using return.

  } // end of last if
  return offset;
} // end of animateShades()

When we call it, we can put smiley.offset as the parameter and the variable that will catch the returned value:

smiley.offset = animateShades(smiley.offset);
This is the functional programming style where we know we are reassigning the value of smiley.offset because that’s what we wanted. The main idea behind this style is that nothing is changed without the programmer wanting it to change explicitly using the assignment operator =.

drawSmiley(smiley);

} // end of draw()

That’s one clean machine. We draw. Then we animate. (Though we could animate and then draw. The difference is slight in this instance.)

Let’s take a look at how we can implement animate() to accomplish this:

function animate(ob) {
  // move shades (functional style)
  ob.offset = animateShades(ob.offset);
  // change direction if at a wall
  bounceOffWalls(ob);
  // add speed to x and y
  ob.x += ob.xSpeed;
  ob.y += ob.ySpeed;
  // prevent glitching at the edge
  keepInBounds(ob);
} // end of animateSmiley()
In addition to the original job of adding the speed to the positions for animate(), we have animateShades(), bounceOffWalls(), and keepInBounds() being called. The function is now more complex, but they are all related to animating the characters on the canvas. They belong together.
Note that we are taking advantage of JavaScripts permissiveness — even though awe doesn’t have an offset and this would be an error in most other languages, but since we don’t try to read any properties of offset — it’s a primitive, after all — JS effectively ignores it and moves on. Later on we’ll see that we must check whether an object exists, but we don’t need to get into that now.
